## 🔍 현황 분석 : 왜 우리는 Git 때문에 고생할까?

명령어를 몰라서가 아닙니다. **"언제, 어떻게"** 써야 하는지에 대한 팀 내 약속이 없기 때문입니다. 
우리의 문제점과 개선 방향을 실무 사례로 정리했습니다.

### 1. 브랜치 전략 : "Main은 성역입니다"
* **As-Is (문제) :** `main` 브랜치에서 바로 코드를 수정하고 커밋함. 
    * **사례 :** A 대리가 로그인 기능을 고치다가 실수로 결제 로직을 건드림
    ➡ 그대로 `main`에 커밋
    ➡ 5분 뒤 결제 서버 장애 발생.

* **To-Be (지향) :** 모든 작업은 독립된 `feature/` 브랜치에서 진행.
    * **예제 :** 1. `main`에서 `feature/login-fix` 브랜치 생성
        2. 작업 완료 후 동료에게 리뷰(PR) 요청
        3. 검증된 코드만 `main`으로 합침 (배포 사고 원천 차단)

### 2. 동기화 방식 : "무지성 Pull은 시한폭탄"
* **As-Is (문제) :** 내 코드가 완성되지 않았는데 일단 `git pull`부터 실행.
    * **사례 :** 서버의 최신 코드와 내 코드가 뒤섞이면서 원치 않는 병합 커밋(Merge commit)이 수십 개 발생
    ➡ 히스토리가 엉망이 되어 나중에 복구 불가능.
* **To-Be (지향) :** `fetch`로 먼저 간을 보고, 상황에 맞춰 병합.
    * **예제 :** 
        ```bash
        git fetch origin          # 서버에 뭐가 올라왔는지 확인만 함
        git log ..origin/main     # 내가 안 가져온 커밋이 5개나 있네? 
        # 내용 확인 후 안전할 때만 병합 진행
        ```

### 3. 충돌 대응 : "덮어쓰기는 협업 거부 선언"
* **As-Is (문제) :** 충돌(Conflict)이 나면 해결하기 귀찮아서 `-f` (force) 옵션으로 밀어버림.
    * **사례 :** B 사원이 충돌을 피하려고 `git push -f` 실행 
    ➡ 어제 밤새 작업한 C 과장의 코드가 서버에서 증발 
    ➡ 팀 분위기 싸해짐.

* **To-Be (지향) :** 수동 해결을 원칙으로 하며, 논리적 흐름에 따라 코드를 합침.
    * **예제 :** 충돌 발생 시 IDE의 충돌 해결 툴을 사용하여 두 코드를 비교 분석 
    ➡ <B>"기능적으로 두 로직이 모두 필요하다면 합치고, 중복이라면 최신 것을 선택"</B>하는 판단 과정을 거침.

### 4. 히스토리 관리 : "태그 없는 배포는 지도 없는 항해"
* **As-Is (문제) :** 커밋 메시지가 "수정", "진짜 최종", "진짜 진짜 최종".
    * **사례 :** 운영 서버에 문제가 생겼는데, 어느 커밋이 배포된 버전인지 몰라서 롤백을 못 함 
    ➡ 복구 시간 지연으로 피해 확산.
* **To-Be (지향) :** 의미 있는 단위로 커밋하고, 배포 시점에 `Tag`를 남김.
    * **예제** 
        ```bash
        # 커밋 메시지는 구체적으로 : "feat : 로그인 API 예외 처리 추가"
        # 배포 시 깃발 꽂기 :
        git tag -a v2.1.0 -m "2026-04-25 정기 배포"
        ```
    * **결과 :** 사고 발생 시 `git switch v2.1.0` 한 줄로 즉시 정상 시점 복구 가능.

        | 구분 | As-Is (현재의 문제) | To-Be (지향할 모습) | 효과 |
        | :--- | :--- | :--- | :--- |
        | **브랜치** | `main` 직접 커밋 | `feature/` 브랜치 분리 | **배포 안정성 확보** |
        | **동기화** | 일단 `pull` 받고 보기 | `fetch` 후 상황 판단 | **코드 오염 방지** |
        | **충돌** | `force push`로 덮기 | 수동 해결 및 논리적 병합 | **동료 코드 보호** |
        | **히스토리** | 추적 불가한 메시지 | 명확한 기록 & Tag 활용 | **장애 대응 속도 향상** |

## 실무 Git 사용 가이드 (상황별 판단 매뉴얼)


### 1. 새로운 기능 개발 시작 (Branch & Switch)
남의 코드를 건드리지 않고 안전한 내 작업 공간을 만드는 과정

* **상황 :** `main` 브랜치에서 `feature/login` 기능을 새로 만들 때
* **실행 순서**
    1.  **상태 확인 :** `git status` (현재 내 위치와 변경 사항 확인)
    2.  **최신화 :** `git pull origin main` (남들이 업데이트한 최신 코드 가져오기)
    3.  **브랜치 생성 및 이동 :** `git switch -c feature/login`
* **Tip :** 브랜치명은 `feature/기능명`, `bugfix/이슈번호` 식으로 규칙을 정하면 관리가 편합니다.

---

### 2. 동료의 작업 확인 (Fetch vs Pull)
무턱대고 합치기 전에 "뭐가 바뀌었나?" 간을 보는 단계입니다.

![Git 충돌 해결 프로세스](../img/git/git_post0002_001.png)

* **상황 :** 팀원이 코드를 올렸는데, 내 코드가 깨질까 봐 걱정될 때
* **권장 패턴**
    ```bash
    git fetch origin          # 서버의 변경사항을 로컬로 가져만 옴 (내 코드 변화 없음)
    git log main..origin/main # 내 main과 서버의 main 차이점 비교
    ```
* **판단 :**  충돌이 없을 것 같다면? `git pull origin main`
    * 위험해 보인다면? 동료와 상의 후 수동으로 합칠 준비를 합니다.

### 3. 히스토리 관리의 핵심 (Merge vs Rebase)
협업 시 가장 많이 논의되는 지점입니다. 아래 기준을 따르세요.

![Git 충돌 해결 프로세스](../img/git/git_post0002_002.png)

**① Merge (안전한 병합)**
* **언제 :** 기능을 완료하고 `main`이나 `develop` 같은 공용 브랜치에 합칠 때.
* **특징 :** "누가 언제 합쳤는지" 기록이 남습니다. 장애 추적에 유리합니다.
    ```bash
    git switch main
    git merge feature/login
    ```

**② Rebase (깔끔한 한 줄 히스토리)**
* **언제 :** 내 **개인 브랜치**에서 작업 중에 `main`의 최신 내용을 내 브랜치에 반영하고 싶을 때.
* **특징 :** 불필요한 merge commit 없이 히스토리가 한 줄로 정렬됩니다.
    ```bash
    git switch feature/login
    git rebase main
    ```
* **⚠️ 주의 :** 이미 서버에 **Push한 커밋**은 절대로 Rebase 하지 마세요. 팀원들의 히스토리가 꼬여서 대참사 발생

### 4. 충돌(Conflict) 발생 시 대응법
충돌은 에러가 아닙니다. Git이 "누구 코드가 맞는지 결정해달라"고 요청하는 것입니다.

1.  **확인 :** `git status`로 충돌 난 파일 확인
2.  **수정 :** IDE(VS Code, IntelliJ 등)에서 충돌 구간 확인
    * `Incoming Change` : 남이 짠 코드
    * `Current Change` : 내가 짠 코드
3.  **결정 :** 하나를 선택하거나, 개발자 간의 상의를 통해 두 코드를 적절히 수정
4.  **완료** 
    ```bash
    git add [파일명]
    git commit -m "fix : resolve conflict"
    ```

---

### 5. 릴리즈의 이정표 (Tag)
장애 발생 시 '어느 시점으로 돌아갈지' 알려주는 포인터 역할

* **상황 :** 운영 서버 배포 시점 (v1.2.0)
* **실행**
    ```bash
    git tag -a v1.2.0 -m "2026-04-24 로그인 기능 배포"
    git push origin v1.2.0
    ```
* **효과 :** 장애 시 `git switch v1.2.0`으로 즉시 당시 코드를 복구 가능

## [실무 매뉴얼] Git 충돌(Conflict) 해결 프로세스

Git에서 충돌은 단순한 에러가 아니라, 서로 다른 두 작업을 하나로 합치기 위한 "의견 조율" 단계입니다.
당황하지 말고 아래의 체계적인 프로세스에 따라 해결하세요.

1. 상황 파악 (Detection) : "무엇이 꼬였는가?"
충돌 메시지(CONFLICT (content) : Merge conflict in...)를 확인했다면, 가장 먼저 영향 범위를 파악해야 합니다.

    ```bash
    # 실행 명령어 
    git status
    ##상태 확인 : Unmerged paths 항목 아래에 both modified라고 표시된 파일들이 현재 충돌이 난 파일들입니다.
    ```

2. 충돌 코드 분석 (Analysis) & 실무 예제
충돌이 발생한 파일을 열면 Git이 충돌 지점에 특수한 마커를 삽입

3. 의미 기준 판단 (Decision) : "무엇이 정답인가?"
정답은 없지만, <B>"전체 시스템 로직에 부합 여부"</B>를 기준으로 한다.

4. 해결 및 확정 (Resolution) : "깔끔하게 정리했는가?"
판단이 끝났으면 코드를 수정하고 마커 삭제

    ```bash
    # 마무리 절차
    # 특수 마커 제거 : <<<<, ====, >>>> 기호들을 한 줄도 남김없이 삭제
    # 스테이징
        git add .
        # or
        git add <파일명>

    # 커밋으로 마무리
        git commit -m "Fix : Resolve merge conflict in login logic"

    # 최종 검증 : 해결 후에는 로컬 환경에서 빌드 및 테스트
    ```

## [ 상황 예시 ] 주문 금액 계산 로직의 충돌
이 사원은 신규 할인 정책(쿠폰)을 반영하라는 지시를 받았고, 정 선임은 전체적인 코드의 안정성을 위해 공통 검증 로직을 강화했습니다. 
두 사람의 코드가 합쳐지면서 충돌이 발생했습니다.

![Git 충돌 해결 프로세스](../img/git/git_post0002_003.png)

**충돌 코드 분석**
```Java
public double calculateOrder(Order order) {
<<<<<<< HEAD (이 사원의 작업)
    // 신규 쿠폰 할인 로직 반영
    double total = order.getBasePrice();
    if (order.hasCoupon()) {
        total -= order.getCouponDiscount();
    }
    return total;
======= (경계선)
    // 계산 전 주문 유효성 검사 강화 (공통 모듈 반영)
    if (order == null || order.getItems().isEmpty()) {
        throw new InvalidOrderException("주문 정보가 올바르지 않습니다.");
    }
    return order.getBasePrice();
>>>>>>> main (정 주임의 작업)
```

1. 의미 기준 판단 (Decision) : "무엇이 정답인가?"
여기서 정 선임이 중재자로 투입되어 두 코드를 분석
    - 이 사원의 코드 분석
    비즈니스 요구사항인 '할인 기능'을 구현했습니다. 필수 기능이지만, 
    주문 객체가 null일 경우 NullPointerException이 발생할 위험 높음

    - 정 선임의 코드 분석
    시스템의 안정성을 위해 '예외 처리'를 강화 체크
    안정적이지만, 새로 도입된 '할인 정책'이 빠져 있습니다.

    - 결론
    정 선임의 안전한 검증 로직을 먼저 실행하고, 
    그 후에 이 사원이 만든 할인 로직이 실행되도록 병합 필요 

2. 해결 및 확정 (Resolution) : "깔끔하게 정리했는가?"
    - 정선임은 두 사람의 의도를 모두 살려 아래와 같이 최종 코드를 확정
        ```Java
        public double calculateOrder(Order order) {
            // 1단계 : 정 주임의 유효성 검사 (안정성 확보)
            if (order == null || order.getItems().isEmpty()) {
                throw new InvalidOrderException("주문 정보가 올바르지 않습니다.");
            }

            double total = order.getBasePrice();

            // 2단계 : 이 사원의 할인 로직 (요구사항 반영)
            if (order.hasCoupon()) {
                total -= order.getCouponDiscount();
            }

            return total;
        }
        ```

### 판단 및 해결 과정
- 이 사원(주니어)
    내 기능을 넣는 것에만 급급하지 않았는지, <B>기존 시스템의 예외 케이스를 놓치지는 않았는지 점검 필요</B>
- 정 선임(시니어)
    코드의 안정성을 높이는 것도 중요하지만, <B>다른 팀원이 개발 중인 신규 기능이 누락되지 않도록 소통</B>해야 합니다.
- 김 책임(중재자)
    "누가 맞느냐"가 아니라 <B>"어떤 순서로 코드가 흐르는 것이 시스템적으로 안전하고 정확한가"</B>를 기준으로 병합을 주도합니다.

## 팀 Git 협업 및 브랜치 관리 가이드

이 문서는 팀 내 코드 품질 유지와 원활한 협업을 위해 반드시 준수해야 할 4가지 핵심 원칙을 정의
&nbsp;&nbsp;&nbsp;<B>※ 서로의 작업 흐름을 존중하고 장애를 예방하기 위한 약속</B>


### 1. 브랜치 전략 강제 (Branch Naming Convention)
브랜치 이름만 보고도 작업의 성격과 목적을 즉시 파악할 수 있어야 합니다. 모든 브랜치는 반드시 아래의 접두사(Prefix)를 사용합니다.

* **`feature/`** : 새로운 기능 개발 작업  
    * 예 : `feature/user-login`, `feature/payment-api`
* **`bugfix/`** : 개발 단계나 QA 중 발견된 결함 수정  
    * 예 : `bugfix/validation-error`, `bugfix/issue-45`
* **`hotfix/`** : 운영 환경(Production)에서 발생한 긴급 장애 대응  
    * 예 : `hotfix/login-crash`, `hotfix/db-connection-leak`
* **`docs/`** : 코드 변경 없는 문서 수정 (README, API 명세 등)  
    * 예 : `docs/update-readme`

### 2. 커밋 메시지 품질 (Commit Message Standards)
커밋 메시지는 동료 개발자가 내 코드를 읽기 전에 거치는 <B>'설명서'</B>입니다. 무의미한 메시지는 코드 리뷰의 효율성 저하

**✅ 권장 형식**
`[Type] 작업 요약 (이슈 번호)`

* **`[Feat]`** : 새로운 기능 추가
* **`[Fix]`** : 버그 수정
* **`[Refactor]`** : 코드 리팩토링 (기능 변경 없이 가독성 / 구조 개선)
* **`[Style]`** : 코드 포맷팅, 세미콜론 누락 수정 (로직 변경 없음)
* **`[Chore]`** : 빌드 스크립트 수정, 패키지 매니저 설정 등

**❌ 지양해야 할 예시**
* `수정`, `작업 중`, `...` (내용 파악 불가)
* `진짜 최종 수정`, `제발 되라` (히스토리 오염)

### 3. Force Push 엄격 금지 (No Force Push)
원격 저장소의 히스토리를 강제로 덮어쓰는 `git push --force`는 팀 전체의 작업 흐름을 파괴할 수 있습니다.

* **위험성** : 다른 팀원이 이미 해당 브랜치에서 작업 중인 경우, 그들의 작업 내역이 증발하거나 타임라인이 꼬여 복구에 막대한 시간이 소요됩니다.
* **금지 규칙** : **`main`**, **`develop`** 등 모든 공용 브랜치에서의 Force Push는 원칙적으로 금지합니다.
* **예외** : 자신의 개인 feature 브랜치에서 `rebase` 후 커밋을 정리(Squash)하여 PR을 올리기 직전에만 제한적으로 허용합니다.

### 4. PR 전 동기화 및 충돌 해결 (Pre-PR Sync)
Pull Request(PR)를 올리기 전, 작업 브랜치를 최신 상태로 유지하는 것은 작업자의 **기본적인 책임**입니다.

![Git 충돌 해결 프로세스](../img/git/git_post0002_004.png)

**🛠 작업 프로세스 (Sync Workflow)**
1.  **목적지 브랜치 최신화** 
    `git switch main` ➡ `git pull origin main`
2.  **내 브랜치로 병합 / 재정렬**
    `git switch feature/my-task` ➡ `git merge main` (또는 `git rebase main`)
3.  **로컬 충돌 해결** 
    충돌(Conflict)이 발생하면 작업자가 직접 코드를 검토하여 해결하고 테스트를 마칩니다.
4.  **최종 Push 및 PR 생성**
    충돌이 해결된 깨끗한 상태의 코드를 원격에 Push 합니다.

## Git 셋팅 및 참고 사항

### 1. 환경 설정 및 기초 Setup
본인 계정 정보가 맞는지 꼭 확인하세요. 엉뚱한 이름으로 커밋이 남으면 추적이 어렵습니다.

**계정 설정**
```bash
git config --global user.name "Your Name"
git config --global user.email "your-email@company.com"
```

**브랜치 및 파일 조작**
```bash
git switch [브랜치명]          # 브랜치 이동 (checkout 대신 권장)
git switch -c [브랜치명]       # 브랜치 생성 및 이동
git restore [파일명]           # 작업 중인 파일 변경사항 취소
```

### 2. 표준 워크플로우
새로운 기능을 개발하거나 버그를 고칠 때는 반드시 이 순서를 지켜주세요.

**Step 1. 작업 시작 (Sync & Branch)**
내 코드를 짜기 전에 원격 저장소(origin)의 최신 상태를 가져옵니다.

```bash
git switch main
git pull origin main
git switch -c feature/login-api  # 규칙 : feature/기능명| bugfix/이슈번호
```

**Step 2. 작업 및 저장 (Commit)**
커밋은 "의미 있는 단위"로 쪼개서 자주 하는 것이 좋습니다.

```bash
git status              # 변경된 파일 목록 확인 (필수 습관)
git add .               # 전체 변경사항 스테이징
git commit -m "[Feat] 로그인 유효성 검사 로직 추가"
```

**Step 3. 원격 업로드 및 PR (Push)**
```bash
git push origin feature/login-api
# 이후 GitHub/GitLab에서 PR(Pull Request) 생성
```

### 3. 핵심 규칙 및 금지 사항
- **git push --force 금지**
    공용 브랜치(main | develop) 히스토리를 절대 덮어쓰지 마세요.
- **커밋 메시지 품질 유지**
    "수정" | "asdf" 등은 반려 대상입니다. [Feat] | [Fix] | [Docs] 등 접두사를 활용하세요.
- **PR 전 동기화**
    PR을 올리기 전 반드시 main의 최신 내용을 내 브랜치에 합쳐서 충돌 여부를 먼저 확인하세요.
- **민감 정보 업로드 금지 (주의사항 : ★★★★★)**
    API Key | DB PW 등이 .gitignore에 포함되었는지 확인하세요.
