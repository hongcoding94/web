## 1단계: [진단] 서버의 기초 체력 점검 (시스템 자원)
**상황:** "서버 응답이 갑자기 느려졌어요!"라는 보고가 들어왔을 때 가장 먼저 수행합니다.

| 명령어 | 주요 옵션 및 설명 | 실무 활용 팁 (Senior's Choice) |
| :--- | :--- | :--- |
| **`top` / `htop`** | CPU/메모리 점유율 실시간 모니터링 | 자원 점유 1위 프로세스를 즉시 식별하여 부하 원인 파악 |
| **`free -m`** | 가용 메모리 양 확인 (MB 단위) | OOM(Out Of Memory) 발생 가능성을 사전에 진단 |
| **`df -h`** | 디스크 여유 공간 확인 (GB/MB 단위) | 로그 과다 누적으로 인한 Disk Full(100%) 상태 점검 |
| **`du -sh *`** | 디렉토리별 용량 요약 확인 | `du -sh * \| sort -hr`로 용량 잡아먹는 범인 검거 |

<br/>

### 기초 체력 진단
*   **시스템 전체 부하 확인:** 현재 CPU를 가장 많이 쓰는 '범인(프로세스)' 찾기에 좋습니다.
    *   *예시:* `$ htop` 실행 후 상단 `%CPU`가 100%인 프로세스의 PID(번호)를 메모합니다.
*   **메모리 부족(OOM) 여부 확인:** 앱이 자꾸 죽는다면 가용 메모리가 있는지 봅니다.
    *   *예시:* `$ free -m` 실행 시 `available` 수치가 100MB 미만이라면 메모리 누수를 의심할 수 있습니다.
*   **디스크 풀(Full) 점검:** 로그가 꽉 차서 서버가 멈췄는지 확인합니다.
    *   *예시 1:* `$ df -h`로 `/` 경로의 `Use%`가 100%인지 확인합니다.
    *   *예시 2:* `$ du -sh * | sort -hr`을 쳐서 용량이 가장 큰 로그 폴더를 찾아냅니다.

## 2단계: [연결] 통로가 열려 있는가? (네트워크 및 포트)
**상황:** "포트 8080으로 접속이 안 돼요!" (자원은 널널한데 접속만 안 되는 상황)


| 명령어 | 주요 옵션 및 설명 | 실무 활용 팁 (Senior's Choice) |
| :--- | :--- | :--- |
| **`netstat -ntlp`**| 오픈된 포트와 수신(Listen) 프로세스 확인 | 서비스가 정상 포트에서 대기 중인지, IP 바인딩 확인 |
| **`lsof -i:8080`** | 특정 포트(8080)를 점유한 PID 확인 | 포트 충돌 방지 및 실행 중인 앱의 정상 구동 판단 |
| **`ping` / `telnet`** | 기초 연결 및 특정 포트 통로 확인 | 상대 서버 생존 여부 및 특정 포트(DB 등) 개방 체크 |
| **`curl -v`** | API 요청 테스트 및 상세 헤더 분석 | 화이트리스트 차단 여부나 HTTP 응답 코드 분석 |

<br/>

### 서비스 통로 확인
*   **포트 리스닝 상태 점검:** 내 서버가 외부 요청을 받을 준비가 됐는지 확인합니다.
    *   *예시:* `$ netstat -ntlp | grep 8080` 실행 시 결과가 아무것도 없다면 앱이 죽은 것입니다.
*   **포트 점유자 식별:** 누가 이 포트를 쓰고 있는지 정확히 찾아냅니다.
    *   *예시:* `$ lsof -i:8080`으로 8080을 점유 중인 실제 PID와 프로세스 명을 확인합니다.
*   **외부 통신 및 권한 분석:** 방화벽이나 화이트리스트에 막혔는지 테스트합니다.
    *   *예시 1:* `$ curl -v [http://target-api.com](http://target-api.com)`을 날려 HTTP 403 에러가 뜨면 화이트리스트 차단입니다.
    *   *예시 2:* `$ telnet db-server 5432`로 PostgreSQL DB 포트가 열려 있는지 찔러봅니다.

## 3단계: [분석] 왜 아픈 것인가? (로그 및 경로 심층 분석)
**상황:** "접속은 되는데 데이터 처리가 안 되거나 에러가 나요! or 통신에서 에러가 나요!"

| 명령어 | 주요 옵션 및 설명 | 실무 활용 팁 (Senior's Choice) |
| :--- | :--- | :--- |
| **`tail -F`** | 실시간 로그 스트리밍 (갱신 추적) | 로그 로테이션(파일 바뀜) 시에도 추적 유지 |
| **`grep`** | 특정 키워드 및 문맥 검색 | `grep -i -A 5 -B 5 "ERROR"`로 전후 맥락 파악 |
| **`traceroute`** | 목적지까지의 경로 추적 | `$ traceroute -n [IP]`로 구간별 지연 및 방화벽 차단 지점 확인 |
| **`mtr`** | 실시간 경로 + 유실률 확인 | `$ mtr -r [IP]`로 어느 구간에서 패킷이 드랍되는지 즉시 파악 |
| **`diff -u`** | 파일 간 차이점 비교 | 설정 파일 수정 전후 비교로 휴먼 에러 방지 |

<br/>

### 원인 심층 추적
*   **실시간 에러 포착 :** 서버 내부의 에러를 실시간으로 확인 가능
    * *예시:* `$ tail -F /logs/app.log | grep --line-buffered "ERROR"`를 띄워두고 에러가 발생하는 순간을 지켜봅니다.
*   **문맥 파악 :** 에러가 왜 났는지 앞뒤 상황(Stack Trace)을 확인이 가능
    * *예시:* `$ grep -i -A 5 -B 5 "NullPointerException" app.log`로 에러 발생 전후 5줄을 함께 봅니다.
*   **네트워크 구간 지연 분석 :** 외부 API가 느린 게 우리 서버 문제인지, 통신사 구간 문제인지 봅니다.
    * [중요] 서비스 업체는 통신 장애 발생 여부에 따라 패널티가 적용되므로,  
    연동 구간의 통신 흐름과 장애 지점을 정확히 이해해야 합니다.
    * *예시:* `$ mtr 8.8.8.8`을 실행해 특정 구간에서 패킷 손실(Loss %)이 발생하는지 실시간으로 봅니다.
*   **내부망인가 외부망인가?**
    * `traceroute` 결과에서 첫 1~3번째 Hop(사설 IP 구간)에서 지연이 있다면 우리 집/회사 공유기나 스위치 문제.
    * 그 이후 공인 IP 구간에서 지연이 있다면 통신사(ISP) 혹은 목적지 서버의 네트워크 문제로 규정.
*   **방화벽 구간 통과 여부**
    * 특정 지점부터 `* * *`만 나온다면 해당 라우터에서 보안 정책상 ICMP/UDP 패킷을 드랍하는 것인지,  
    아니면 실제로 죽은 것인지 `-I`(ICMP)와 `-T`(TCP) 옵션을 바꿔가며 테스트.

## 4단계: [조치] 어떻게 고칠 것인가? (프로세스 제어 및 복구)
**상황:** "원인을 찾았으니 이제 빠르게 정상화해야 합니다!"

| 명령어 | 주요 옵션 및 설명 | 실무 활용 팁 (Senior's Choice) |
| :--- | :--- | :--- |
| **`kill -9 [PID]`** | 프로세스 강제 종료 | 정상 종료되지 않는 좀비 프로세스 제거 |
| **`chmod / chown`** | 파일 권한 및 소유자 변경 | 실행 권한 부여 및 "Permission Denied" 해결 |
| **`tar -cvzf`** | 파일/디렉토리 압축 백업 | 조치 전 중요한 데이터나 로그를 안전하게 보관 |

<br/>

### 긴급 복구 및 제어
*   **강제 종료:** 정상적으로 안 꺼지는 앱을 즉시 중단시킵니다.
    *   *예시:* `$ kill -9 1234` (1234는 앞서 찾은 PID)로 좀비 프로세스를 제거합니다.
*   **권한 및 소유권 수정:** "Permission Denied"로 파일 작성이 안 될 때 조치합니다.
    *   *예시:* `$ chmod +x startup.sh`로 실행 권한을 주거나, `$ chown user:user log/`로 소유권을 바꿉니다.
*   **[중요] 작업 전 백업:** 파일을 수정하거나 삭제하기 전에 안전장치를 만듭니다.
    *   *예시:* `$ tar -cvzf log_backup_$(date +%Y%m%d).tar.gz ./logs/`로 현재 로그를 압축 백업합니다.

## 5단계: [효율] 업무 생산성 향상 (관리 및 자동화)
**상황:** "똑같은 명령어를 또 치기 귀찮을 때" 혹은 "빠른 검색이 필요할 때"

| 명령어/기능 | 주요 역할 | 실무 활용 팁 (Senior's Choice) |
| :--- | :--- | :--- |
| **`Ctrl + R`** | 명령어 역방향 검색 | 키워드 입력만으로 히스토리 내 긴 명령어 자동 완성 |
| **`find`** | 파일 및 디렉토리 검색 | `-mtime -1` 옵션으로 최근 수정된 파일 검색 |
| **`alias`** | 명령어 별칭(단축어) 설정 | 자주 쓰는 복잡한 명령어를 단축어로 등록 |

<br/>

### 생산성 향상 꿀팁
*   **과거의 나에게 묻기:** 히스토리를 뒤져서 시간을 아낍니다.
    *   *예시:* `Ctrl + R`을 누르고 `postgresql`이라고만 치면 예전에 썼던 긴 접속 명령어가 자동 완성됩니다.
*   **파일 빨리 찾기:** 설정 파일 위치를 단숨에 찾아냅니다.
    *   *예시:* `$ find /etc -name "*.conf"`로 시스템 설정 파일들을 검색합니다.
*   **단축어 활용:** 손가락의 피로를 줄입니다.
    *   *예시:* `.bashrc`에 `alias logtail='tail -F /var/log/app.log'`를 등록해두고 한 단어로 로그를 확인합니다.

## 실무에서 경험 후에 "한 끗 차이" 기술 요약

*   **복합 명령어:** `kill -9 $(lsof -t -i:8080)` (8080 포트 쓰는 프로세스 식별 후 즉시 처단)
*   **데이터 정합성:** 장애 조치 전후로 `diff -u`를 사용하여 설정 파일의 어떤 부분이 바뀌었는지 반드시 비교하여 휴먼 에러를 방지합니다.
*   **로그 로테이션:** `df -h`로 디스크가 찼을 때, 무작정 지우기보다 `tar`로 압축 후 다른 서버로 옮기거나 백업하는 습관을 만들어햐한다.
