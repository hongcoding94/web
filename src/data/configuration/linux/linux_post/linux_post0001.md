### 1. 파일 및 경로 관리: "길을 잃지 않는 법"
단순히 파일을 만드는 것을 넘어, 권한과 위치를 정확히 파악하는 것이 중요합니다.

*   **`ls -alht`** : 실무에서는 이 조합을 가장 많이 씁니다.
    *   `-h` (human-readable): 파일 용량을 KB, MB 단위로 보기 좋게 표시합니다.
    *   `-t` (time): 최신 수정 시간 순으로 정렬합니다. (방금 수정한 파일을 찾을 때 유용)
*   **`cd -`** : **(꿀팁)** 직전에 있었던 디렉토리로 바로 돌아갑니다. 두 폴더 사이를 왔다 갔다 할 때 환상적입니다.
*   **`mkdir -p`** : `mkdir -p /logs/app/2026/05` 처럼 깊은 경로를 한 번에 생성할 때 씁니다. 배포 스크립트 작성 시 필수입니다.

### 2. 로그 분석의 삼대장: `tail`, `grep`, `less`
서버에 장애가 났을 때 가장 먼저 타이핑하게 될 명령어들입니다.

*   **`tail -f [파일명]`**: 실시간으로 올라오는 로그를 봅니다.
    *   **실무 팁 :** `tail -f app.log | grep --line-buffered "ERROR"` 처럼 사용하면 실시간 로그 중 에러만 골라낼 수 있습니다.
*   **`less`** : `cat`으로 수만 줄의 로그를 출력하면 터미널이 멈출 수 있습니다.
+@ `less`는 메모리를 적게 쓰며 페이지 단위로 보여줍니다.
    *   `Shift + G` : 파일의 맨 끝으로 이동.
    *   `g` : 파일의 맨 앞으로 이동.
*   **`grep` (Global Regular Expression Print)**
    *   `grep -i` : 대소문자 구분 없이 검색.
    *   `grep -A 5 "ERROR"` : 에러 발생 지점 **이후(After)** 5행까지 함께 출력 (Stack Trace 확인용).

### 3. 프로세스와 리소스 관리: "서버가 왜 느리지?"
내 애플리케이션이 메모리를 잡아먹고 있거나, 포트가 점유되어 실행이 안 될 때 사용합니다.

*   **`ps -ef | grep [프로세스명]`** : 내 프로그램의 PID(프로세스 ID)를 찾을 때 씁니다.
*   **`htop`** : 기본 `top`보다 훨씬 시각적입니다. CPU 코어별 점유율과 메모리 사용량을 직관적으로 보여줍니다. (별도 설치 필요)
*   **`kill -9 [PID]`** : 보통은 `kill -15`(정중한 종료 요청)를 먼저 쓰고, 안 들으면 `-9`(강제 종료)를 씁니다.
*   **`df -h` vs `du -sh`**
    *   `df -h`: 디스크 전체 잔여 용량 확인. 
    *   `du -sh *`: 현재 디렉토리 내 폴더별 용량 확인. (어떤 놈이 용량을 다 먹었는지 **범인 찾기용**)

### 4. 네트워크 및 통신: "연결이 왜 안 되지?"
방화벽 문제인지, 서버 미실행 문제인지 판별합니다.

1.   **`netstat -tulpn` or `ss -tulpn`** : 현재 서버에서 열려 있는 포트와 해당 포트를 사용하는 PID를 보여줍니다. 
        *   `8080` 포트가 이미 사용 중이라 서버가 안 뜰 때 범인을 잡는 용도
2.   **`curl -v [URL]`** : `-v`(verbose) 옵션을 주면 HTTP 헤더 정보까지 다 보여줍니다. API 디버깅 시 필수입니다.
3.   **`nc -zv [IP] [PORT]`** : 특정 서버의 포트가 열려 있는지(방화벽 통과 여부) 간단히 체크할 때 씁니다.

### 5. 권한 관리: "Permission Denied 해결"
배포한 실행 파일이 작동하지 않을 때의 90%는 권한 문제입니다.

실무에서 가장 헷갈리기 쉬운 **`chmod`**와 **`chown`**의 핵심을 구조적으로 다시 정리해 드릴게요. 
이 두 명령어의 차이를 명확히 아는 것만으로도 서버 트러블슈팅 능력이 비약적으로 상승합니다.

### 5-1. 한눈에 비교하는 `chmod` vs `chown`
| 구분 | **chmod** (Change Mode) | **chown** (Change Owner) |
| --- | --- | --- |
| **핵심 목적** | **파일에 무엇을 할 수 있는가?** (행위 제한) | **누가 이 파일의 주인인가?** (귀속 변경) |
| **대상 권한** | 읽기(r), 쓰기(w), 실행(x) | 소유 사용자(User), 소유 그룹(Group) |
| **주요 상황** | 스크립트 실행 권한 부여, 보안 설정 | 파일 배포 후 실행 계정 일치 작업 |
| **명령어 예시** | `chmod 755 deploy.sh` | `chown nginx:nginx app.log` |

### 5-2. `chmod`: "이 파일로 무엇을 할 것인가?"
리눅스 권한은 **3진법 숫자(4, 2, 1)**의 합으로 구성된 세 자릿수를 이해하는 것이 핵심입니다.

#### 💡 권한 계산기 (합산 방식)
*   **4 (Read)** : 읽기
*   **2 (Write)** : 쓰기
*   **1 (Execute)** : 실행
*   **예:** `7(4+2+1)`은 모든 권한, `5(4+1)`은 읽기와 실행만 가능.

#### 📂 세 자릿수의 의미
1.  **첫 번째 자리** : 파일 소유자 (Owner)
2.  **두 번째 자리** : 그룹 멤버 (Group)
3.  **세 번째 자리** : 그 외 나머지 (Others)

> **실무 필수 세트:**
> *   **`chmod 755`** : 실행 파일(Shell Script) 등에 사용. (나는 다 하고, 남들은 실행만)
> *   **`chmod 644`** : 일반 설정 파일이나 소스 코드. (나는 수정하고, 남들은 읽기만)
> *   **`chmod 400`** : `.pem` 키 파일 등 극도의 보안이 필요한 파일. (나만 읽기 가능)

### 5-3. `chown`: "누가 이 파일의 관리자인가?"
파일의 주인을 바꾸는 명령어입니다. 주로 Root 권한으로 생성된 파일을 서비스 계정(예: `ubuntu`, `nginx`, `jenkins`)으로 넘겨줄 때 사용합니다.

#### 📝 명령어 문법 가이드
*   **사용자:그룹 변경**: `chown user:group file` (가장 정석적인 방법)
*   **사용자만 변경**: `chown user file`
*   **그룹만 변경**: `chown :group file`

#### 실무 치트키: `-R` 옵션 (Recursive)
배포 시 특정 폴더와 그 안의 수천 개 파일 소유권을 한 번에 바꿔야 할 때 필수입니다.
```shell
sudo chown -R www-data:www-data /var/www/my-project
```
> **Tip**: `-R`을 쓸 때는 경로를 절대 틀리지 않도록 주의하세요! (잘못하면 시스템 전체 소유권이 꼬일 수 있음)

### 5-4. 실무에서 겪어봤던 경험과 문제 해결 방법
1.  **"파일이 안 열려요"** 
$\rightarrow$ `ls -l`로 소유자(`chown`)와 읽기 권한(`chmod`) 확인.  
<br/>
2.  **"스크립트 실행이 안 돼요"** 
$\rightarrow$ `chmod +x [파일명]`으로 실행 권한 추가.
$\rightarrow$ `whoami`와 접속된 그룹이 다른 경우 `chown [user:group file]`으로 수정  
<br/>
3.  **"로그가 안 쌓여요"** 
$\rightarrow$ 애플리케이션 실행 계정이 해당 폴더에 쓰기(`w`) 권한이 있는지 확인.
$\rightarrow$ 실행 파일(`start.sh`)에 (`/dev/null 2>&1`) 확인 필요 있다면 (`1>/dev/null`)로 변경
$\rightarrow$ 로그 경로 오류  
<br/>
4.  **"SSH 접속 키가 위험하대요"** 
$\rightarrow$ `chmod 400 key.pem`으로 권한을 변경

## 3. 실무에서 권한 관련 트러블슈팅 (Scenario)

개발하다 보면 가장 짜증 나는 에러 중 하나가 **"Permission Denied"** 입니다.  
이때 체크리스트는 다음과 같습니다.

### 상황 A: 스크립트 파일(`.sh`)이 실행이 안 될 때
파일이 생성된 직후에는 실행 권한(`x`)이 없는 경우가 많습니다.
*   **해결**: `chmod +x deploy.sh` (숫자 대신 기호를 써서 '실행 권한 추가'만 할 수도 있습니다.)

### 상황 B: 웹 서버(Nginx/Apache)가 파일을 못 읽을 때
파일 소유자가 `root`로 되어 있거나, 권한이 `600`처럼 빡빡하게 걸려 있으면 웹 서버 유저가 접근을 못 합니다.
*   **해결**: `chown -R www-data:www-data /var/www/project`

### 상황 C: 로그 파일에 로그가 안 쌓일 때
애플리케이션을 실행하는 계정이 해당 로그 파일이나 폴더에 쓰기(`w`) 권한이 있는지 확인해야 합니다.
*   **확인**: `ls -l` 명령어로 맨 앞의 권한 문자열(`drwxr-xr-x`)을 확인하세요.

## 요약 비교

| 구분 | `chmod` | `chown` |
| :--- | :--- | :--- |
| **무엇을 바꾸나?** | 할 수 있는 **행위** (r/w/x) | 파일의 **주인** (User/Group) |
| **언제 쓰나?** | "이 파일을 실행 가능하게 만들고 싶어" | "이 파일의 주인을 nginx로 바꾸고 싶어" |
| **핵심 옵션** | `755`, `644`, `+x` | `-R` (하위 포함), `user:group` |

**⚠️ 주의사항:** 
실무에서 귀찮다고 `chmod 777`을 남발하는 ~~**훗날 보안팀 혹은 해커에게 지옥을 보실수 있습니다.**~~
누구나 수정하고 삭제할 수 있게 되어 보안상 매우 위험히며, 항상 **필요한 최소한의 권한만 주는 것이 원칙**입니다.

### 6. 치트키: 명령어 조합 (Piping & Redirection)
리눅스의 진정한 힘은 명령어들을 엮는 데서 나옵니다.

1.   **`|` (Pipe)** : 앞 명령어의 결과물을 뒤로 넘김.  
        *   예 : `history | grep "docker"` (내가 예전에 쳤던 도커 명령어 찾기)
2.   **`>` vs `>>` (Redirection)**  
        *   `>` : 기존 내용을 지우고 새로 씀 (Overwrite).
        *   `>>` : 기존 내용 끝에 추가함 (Append).  
        *   예 : `node app.js >> app.log 2>&1 &` (로그를 파일에 쌓으면서 백그라운드 실행)

---

### 실무자의 팁

1.  **`history`** : 내가 이전에 입력했던 명령어들을 쭉 보여줍니다. `!번호`를 입력하면 해당 번호의 명령어가 재실행됩니다.
2.  **`alias`** : 긴 명령어를 줄여 씁니다. 
    *   예 : `alias ll='ls -alht'` 를 `.bashrc`에 등록해두면 삶의 질이 올라갑니다.
3.  **`tab` 키 활용** : 파일명이나 경로를 끝까지 치지 마세요. `Tab`을 누르면 자동 완성됩니다. (두 번 누르면 후보군 출력)
4.  **서버 `timeout=0`** : 타임아웃 비활성화 (요청 종료 조건 미설정)
5.  **`Ctrl + C` vs `Ctrl + Z`** 
    *   `Ctrl + C` : 프로세스 완전 종료.
    *   `Ctrl + Z` : 프로세스 일시 중지 (나중에 `fg`로 복구 가능).
