# 배경

<aside>
💡

Quartz Scheduler 기반 배치 시스템에서 특정 시점에 트리거 지연 및 미실행 현상이 간헐적으로 발생하였다.

초기에는 JVM 자원 문제(Heap, GC, CPU)로 의심되었으나, 분석 결과 애플리케이션 자원 고갈 징후는 발견되지 않았다.

이후 심층 분석을 통해 서버 OS 시간과 Quartz 내부 스케줄 기준 시간 간 불일치 현상이 존재함을 확인하였으며,

원인은 고객사 운영 정책에 따른 NTP 시간 동기화 방식에 기인한 것으로 판단하였다.

# 분석

### 1️⃣ 원인 분석

- OS 시간 보정(NTP 동기화) 과정에서 시간 이동(step) 발생
- Quartz Scheduler의 Cron 기반 트리거 계산 로직과 시간 기준 불일치 발생
- 특정 Job이 misfire 처리 없이 skip되는 현상 확인

### 2️⃣ 제약 조건

- NTP 정책 변경은 고객사 정책 영역
- OS 시간 동기화 방식 직접 제어 불가
- 애플리케이션 레벨에서 시간 이동 자체를 차단할 수 없음

→ 근본 원인 제거 방식은 구조적으로 제한적

### 3️⃣ 대응 전략

Watchdog 외부 솔루션을 사용하지 않고, **내부 로그 기반 자가 모니터링 및 자동 복구 방식으로 설계**

### 동작 구조

1. Quartz 실행 시각 로그 기록
2. OS 현재 시간과 마지막 실행 시간 비교
3. 지연 시간 산출
4. 임계치 초과 시 OS 서비스/데몬 재기동 수행

### 임계 조건

| 조건 | 조치 |
| --- | --- |
| 5분 초과 | 경고 로그 기록 |
| 10분 초과 | 서비스/데몬 자동 재기동 |

### 4️⃣ OS별 적용 방식

### ■ Linux

- 로그 기반 실행 지연 감지
- daemon 프로세스 상태 확인
- `systemctl restart` 명령을 통한 재기동

### ■ Windows

- Quartz 실행 로그 파일 기반 지연 감지
- PowerShell 스크립트를 통한 로그 파싱
- `Restart-Service` 명령을 이용한 Windows Service 재기동
- 스케줄러(Task Scheduler) 또는 주기 실행 방식으로 감시 수행

### 5️⃣ 적용 결과

| 구분 | 결과 |
| --- | --- |
| 업체 A | 정책상 적용 불가 |
| 업체 B | 적용 완료 후 동일 유형 장애 미재발 |

적용 환경에서 정상 복구 동작 확인.

# 결론

본 이슈는 애플리케이션 결함이 아닌 **운영체제 시간 동기화 정책(NTP)에 따른 논리적 스케줄 오류**로 정의하였다.

근본 원인 제거가 불가능한 환경을 고려하여 운영 통제 기반 자동 복구 메커니즘을 설계 및 구현하였다.

적용 고객사에서는 서비스 연속성이 확보되었으며, 동일 유형 장애 재발이 발생하지 않았다.

→ 제어 불가능한 영역을 전제로 안정성을 확보한 운영 보완 설계 사례로 판단된다.

### 효율성 공식

| 구분 | 계산 방식 |
| --- | --- |
| 시간 기반 효율성 | 업무 처리 시간 대비 완료 건수
`효율성 = (완료된 작업 수[산출량] ÷ 총 소요 시간)` |
| 비용 기반 효율성 | 산출물 1단위당 비용
`효율성 = (총 Output[산출량] ÷ 총 비용)` |
| 자원 기반 효율성 | CPU, 메모리 사용량 대비 처리량
`효율성 = (처리량[산출량] ÷ 자원 사용량)` |
| 목표 기반 효율성 | 실제 성과 / 목표 성과
`효율성 = (실제 달성 값 ÷ 목표 값) × 100%` |
| **정확성** | 비즈니스 품질
`정확성 = (정답/정확결과 수 ÷ 전체 결과 수 ) × 100%` |
